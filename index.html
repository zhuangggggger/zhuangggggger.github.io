SOLID原则的一点理解

发表于 2021-07-14 更新于 2021-07-18 分类于 计算机

SOLID原则

单一职责原则

一个类或者模块应该有且只有一个改变的原因。

单一职责是分离关注点的结果，关注点分离的越准确，代码也就更加“高内聚，低耦合”，当业务需求变化时候，仅需修改业务关注点的模块，该需的改变是这个模块(类)变化的原因，其他模块稳定不受影响。关注点分离粒度越细，职责也就越单一。

如果仅从功能上看，功能组成的粒度几乎可以无限拆分，类似于宏观到微观。所以应该按业务关注点划分粒度，在业务可能变化点上做到单一职责，稳定的模块也就越多。例如OSI将计算机网络体系结构划分为七层,但实际常用却是TCP/IP四层模型，因为平时业务中四层模型已经可以满足大部分场景。

开放封闭原则

软件实体(类、模块、函数)应该对扩展开放，对修改封闭。

通俗讲是，新的需求，不需要修改旧代码，仅增加新的代码。单一职责原则描述了模块（类）的定义封装准则，开放封闭原则需要考虑代码设计实现，留下扩展点，涉及多态、面向接口编程等。

实践体会：
场景：监听不同主题的消息，处理不同的业务
实现：使用策略模式，定义不同消息主题模块，实现消息处理接口，使用工厂模式根据监听到消息主题创建不同的消息模块，利用多态运行相应的处理算法。
从消息代码结构上看已经留下了扩展点，新的主题由新定义的模块完成业务，但由于旧主题模块依赖的其中一个小模块职责不够单一，新的主题消息需要该模块部分逻辑变化。如果重新定义个模块完成新主题消息处理，存在部分代码逻辑重复，不太好。另一个方法是，重构该模块，将稳定部分和改变部分分离出来形成更小的模块。严格来说这样还是修改了旧代码，我们需要对重构后的模块进行测试，还是增加了工作量。如果我们一开始就将该模块职责定义的够单一，就可以避免这个问题。

这算不算违背开放封闭原则？可能理解不太准确，先记录下

接口隔离原则

客户（client）不应被迫使用对其而言无用的方法或功能。

接口并非越全越好，仅暴露调用者需要的接口就好。避免调用者依赖无用接口而影响稳定性。

实践体会：
从上往下编写代码，先写顶层模块实现，再编写依赖的模块。编写顶层代码时可以清晰感受到，我们需要哪些接口及怎样的接口才能优雅地去实现该逻辑。结合单一职责原则，在关注点分离准确的情况下，接口职责单一，避免后续需要拆分接口。如果是业务需要粗粒度接口，比如前端使用，可以利用门面模式组合细粒度接口，减少请求次数。也可以通过接口隔离原则反过来判断接口是否符合单一职责原则，如果调用者仅需用部分接口，说明提供该组接口的模块不够单一；如果调用者仅使用该接口部分功能，说明该接口职责不够单一。

依赖倒置原则

1.高层次的模块不应该依赖于低层次的模块，两者都应该依赖于抽象接口。
2.抽象接口不应该依赖于具体实现。而具体实现则应该依赖于抽象接口。

通过抽象接口将高层模块和低层模块解耦，高层模块调用抽象接口，低层模块实现抽象接口。抽象接口的定义不应该根据模块实现方法定义，而是模块需要实现(或使用)该抽象接口。

实践体会：
依赖倒置好处在于低层模块变化时，仅需注入抽象接口依赖的新模块，高层模块无需任何改动。例如低层模块为db且要更换db时；低层模块为其他服务接口，单元测试时可mock该接口，返回期望的数据。

里氏替换替换原则

派生类（子类）对象可以在程序中代替其基类（超类）对象。

里氏替换替换原则要求子类型需要满足替换父类型而功能不变。也就是说子类型尽量只做扩展，避免修改父类型，如需重写父类型方法，需遵守父类型约束，保证功能不变。

疑问：里氏替换替换原则是否适合鸭子类型？
项目中遇到一种情况，由于架构和业务的变化，原有的消息收发接口需要根据key做消息转发功能。改动思路为构建新消息模块内嵌原有消息模块，重写接受消息接口增加转发功能（golang实现）。业务层稳定无需改变，仅需注入新依赖的模块，也没有改写旧代码。
golang中没有继承概念，如果内嵌可以看着继承，那么这个新模块并不满足里氏替换替换原则，原有的接口功能已重写，单元测试也无法通过。内嵌只是为了语法方便，组合也可以完成。保留疑问，后续有新的理解再更新。

缓存更新策略

发表于 2021-07-06 更新于 2021-07-14 分类于 计算机

缓存更新策略

Cache Aside

更新数据时先更新数据库，然后让缓存失效。查询时先查缓存，没有命中到则从数据库读取，并更新到缓存中。

分析：
假设读和写并发操作且缓存没有命中情况下
读操作：
1.查询缓存没有命中，先去查数据库
2.更新查数据库得到的数据到缓存
写操作：
a.更新数据库
b.使缓存失效
由于上述的读操作和写操作都不是原子操作，拆分后可能存在顺序：
1.查询缓存没有命中，先去查数据库
a.更新数据库
b.使缓存失效
2.更新查数据库得到的数据到缓存
这种情况下缓存中的数据属于脏数据，如果读写操作上锁（读写锁，互斥锁更不能考虑）写锁持有期间就失去了并发性，同时也不能确保数据强一致性。如果引入Paxos、Raft协议保证一致性，系统会复杂很多很多。考虑到一般情况下数据库查询要比写操作快，实际上要符合缓存没有命中且读写操作为上面顺序概率很小，所以该方案较常用，另外由于有概率出现脏数据，缓存需要设定失效时间，为了防止缓存在同一个时刻同时失效造成缓存雪崩，通常在原有失效时间增加随机值。

网上有些资料提到使用2PC保证一致性，我理解中的2PC、3PC同样可能存在数据不一致异常，有点疑惑先记录下。
2PC在网络异常或者协调者崩溃情况下可能出现数据不一致，但这属于服务非正常的情况下，在服务正常可用情况下2PC是可以保证强一致性的，不过2PC同样存在并发性能问题。

思考：
1.更新数据库后使缓存失效会不会缓存击穿：
处理方法同缓存击穿一样，并发读操作可通过互斥锁防止都请求到数据库，例如redis中SETNX。
2.为什么不更新数据库后同时更新缓存：
在读操作中再更新缓存有点懒汉式的意思，延时加载节省资源。更新数据库后同时更新缓存出现（更a到数据库，更新b到数据库，更新b到缓存，更新a到缓存）情况概率比使缓存失效方案（缓存覆盖热点key）更高。一般情况下Cache Aside方案更适用。

Read/Write Through

Read/Write Through模式由缓存代理了对数据库的操作，对应用服务来说可以把缓存和数据库看着一个整体服务。Read Through模式缓存失效时缓存服务自动从数据库加载。Write Through模式更新数据时如命中缓存则更新缓存再更新数据库，否则直接更新数据库。这里的并发问题由缓存服务保证原子性。

Write-behind（Write Back）

Write-behind模式更新数据时仅更新缓存，后续异步统一批量更新到数据库，减少操作次数提高性能。
mysql InnoDB中redo log也是这种模式，先将操作记录写到redo log，后续再更新到磁盘，减少磁盘io操作。

总结

不同的场景下需求不一样，适合的才是最好的。在能接受范围下的用极低的异常概率换取开发效率和降低维护复杂性也是很划算的。

浮点数精度损失和范围

发表于 2021-07-05 更新于 2021-07-14 分类于 计算机

浮点数

二进制小数表示方法

十进制下表示：

eg: $10.24 = 1 \times 10^1 + 0 \times 10^0 + 2 \times 10^{-1} + 4 \times 10^{-2}$

二进制同理：

eg: $10.24_2 = 1 \times 2^1 + 0 \times 2^0 + 2 \times 2^{-1} + 4 \times 2^{-2}$

也就是小数的二进制只能表示可以写成$x \times 2^y$的数，其他值只能被近似表示，例如$\frac{1}{3}、\frac{5}{7}$。这也是浮点数精度损失原因之一。

IEEE754浮点数

《深入理解计算机系统》:


浮点数的表示有点类似科学计数法，存储结构分为：符号位+指数位+尾数位

规格化的值可以调整阶码E，使尾数 $1 \geq M < 2$，类似十进制中100可以表示为$10 \times 10^1$ 也可以表示为$1 \times 10^2$，第一位永远是1没必要再存了，那么就可以省出空间存更精细的尾数，类科学计数法的存储方式使得32位的float范围比32位的int范围要大

科学计数法是带精确度的，浮点数尾数位类似，如果有10000000000…0001这样的数，尾数位无法全部存储,但在指数位能表示范围内，那么就算没有小数，精度也会丢失。

关于Context(上下文)的理解

发表于 2021-07-03 更新于 2021-07-06 分类于 计算机

Context

In computer science, a task context is the minimal set of data used by a task (which may be a process, thread, or fiber) that must be saved to allow a task to be interrupted, and later continued from the same point. — 维基百科

函数调用

普通函数调用时涉及到调用栈、PC寄存器（程序计数器）BP寄存器（栈基址）和SP寄存器（栈指针）等，函数调用时将返回地址、栈基址和栈指针压入栈中，执行完成再弹出到PC、BP和SP中。将寄存器等处理器运行所需要的状态暂存，并在执行完后恢复状态，也就是cpu的上下文切换。裸机下硬件中断处理程序同理。裸机或各类操作系统中普通函数调用都是一样的，不同架构的cpu，寄存器等硬件结构存在差异，但这一点大概原理应该是一致的。

系统调用

系统调用顾名思义是跑在操作系统上，没有裸机之说法，和普通函数调用最大不同是普通函数运行在用户态，系统调用运行在内核态，通常需要做用户态/内核态切换，为确保内核安全需要严格检查系统调用的序号、参数(特别是指针)等合法性。系统调用涉及到了系统，相比函数调用需要更多额外的开销。

用户态线程（协程）

用户线程在用户空间实现，由用户程序自己实现调度、上下文切换等。内核没有直接对用户线程进行调度，用户线程切换无需切换用户态/内核态，共享进程虚拟内存、全局变量等资源。

Golang runtime中GPM
G：runtime调度的最基本单位，存储着goroutine的执行stack、状态以及任务函数等。
P：P表示逻辑processor，存储着线程M的执行的上下文。
M：计算资源（系统线程)
GPM的理解：向系统动态申请线程，在用户空间中以线程池方式实现调度管理，由于M(系统线程)由系统调度，无法确保一直在工作，P作为概念中的逻辑CPU，在某一个M阻塞时可以结合其他非阻塞M以保持一直在工作的假象。P保存有M的上下文信息，结合其他M即可继续当前任务。G则是用户任务载体，挂在P上运行（与系统线程调度类似，线程挂到物理CPU运行，G挂到P(逻辑CPU)运行）。 P将M和G解耦，实现M和G双向调度管理，以提高效率。

内核态线程

内核态线程也就是常说的线程，同一个进程的线程切换，由于虚拟内存、全局变量等资源共享，无需切换保存，只需切换线程私有资源，比如栈和寄存器。线程由系统调度，涉及系统调用，通常存在用户/内核态切换。
非同一个进程线程切换，同进程切换一样。

进程

进程的切换只能发生在内核态，进程上下文切换包括CPU寄存器、用户/内核堆栈、全局变量和虚拟内存页表、进程表、已打开文件信息的文件表等内核数据。特别是虚拟内存切换后可能缺页，需替换虚拟页（SRAM->DRAM->磁盘，多级页表），性能消耗很大。

总结

无论是普通函数调用、用户/内核态、协程、线程或是进程切换、都需要保存当前执行环境必要信息，以确保后续可以在同一个点重新执行。不同点在于随着概念(或者说直接依赖的运行环境)的复杂化，所需保存的信息更加复杂，性能消耗也更大。
